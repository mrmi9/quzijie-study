> 状态基线：2026-07-13
> 最终目标：完成真实后端、微信登录、生产部署、合规备案、微信审核并正式投入使用。

## 1. 当前结论

当前项目已达到“完整 Mock 演示版”水平，但尚未达到可供真实用户使用的生产状态。

现阶段不应继续扩展排行榜、评论、社交、搜索等非核心功能。下一阶段应从前端功能开发转入生产化建设，优先完成代码基线、后端、数据库和真实微信登录。

当前最优先的工作顺序为：

1. 整理、提交并推送当前 Mock 全功能版本，形成稳定基线。
2. 创建真实后端工程和数据库 Schema。
3. 接通微信登录，并完成 C/C++ 的第一条真实 API 业务闭环。
4. 逐步将错题、收藏、全局统计和 408 从 Mock 切换到真实 API。
5. 完成题库复核、真机验收、部署、备案和微信审核。

## 4. 后端与数据库建设

建议建立独立后端工程，并至少划分开发、测试和生产三套环境。数据库可使用 PostgreSQL 或 MySQL，后端技术栈由团队统一确定。

### 4.1 第一条纵向闭环

不要一次性实现所有接口。第一阶段先使用 C/C++ 打通以下真实链路：

```text
微信登录
→ 获取学科和章节
→ 创建练习
→ 提交答案
→ 完成练习
→ 查看结果
→ 关闭并重新进入后恢复进度
```

该链路稳定后，再复用通用模型启用其余六个学科。

### 4.2 建议数据库模型

- `users`：用户和微信身份。
- `user_sessions` 或 `refresh_tokens`：登录会话。
- `subjects`：学科。
- `chapters`：章节。
- `questions`：题目当前版本。
- `question_versions`：题目历史版本。
- `question_options`：选项。
- `practice_sessions`：普通练习会话。
- `practice_session_questions`：会话题目和题目快照。
- `practice_answers`：逐题答案及幂等 ID。
- `wrong_question_records`：错题、掌握状态和错误次数。
- `favorites`：收藏记录。
- `exams`：408 试卷。
- `exam_questions`：试卷题目及版本快照。
- `exam_drafts`：考试草稿。
- `exam_results`：考试结果和四科统计。
- `audit_logs`：必要的安全和业务审计记录。

### 4.3 后端基础设施

- 数据库迁移脚本。
- 500 道题库导入程序。
- `/health` 和必要的健康检查。
- 统一错误码和响应格式。
- 请求参数 Schema 校验。
- 结构化日志和请求追踪 ID。
- 开发、测试、生产环境配置。
- 密钥和密码的安全配置机制。
- 后端单元测试和 API 集成测试。

### 4.4 关键业务规则

- 正确答案只能由服务端判定。
- 未提交普通题和未交卷 408 试题不得返回答案或解析。
- 每个答题请求使用 `clientAnswerId` 保证幂等。
- 已完成会话和试卷不得重复计分。
- 408 剩余时间以服务端 `expiresAt` 为准。
- 408 到期时由服务端执行幂等自动交卷。
- 题库更新不得改变历史成绩和历史解析。
- 所有用户数据必须按当前登录用户隔离。
- 同一用户只能继续一个普通练习会话。

## 5. 真实微信登录

当前登录页在 Mock 模式下使用演示 Token，在 API 模式下尚未完成 `code` 换取业务 Token 的公共登录能力。

### 5.1 登录流程

1. 小程序调用 `wx.login` 获取临时 `code`。
2. 小程序把 `code` 发送给自己的后端。
3. 后端使用 AppID 和 AppSecret 调用微信 `code2Session`。
4. 后端获取 OpenID，并创建或查找业务用户。
5. 后端签发自己的访问令牌和可选刷新令牌。
6. 小程序保存访问令牌并返回原入口。
7. Token 失效后执行刷新或重新登录。

AppSecret 只能保存在服务端，不能写入小程序源码、Git 仓库或前端配置。

### 5.2 建议新增接口

- `POST /api/v1/auth/wechat/login`
- `POST /api/v1/auth/refresh`
- `POST /api/v1/auth/logout`
- `GET /api/v1/users/me`
- `DELETE /api/v1/users/me` 或等价的数据删除/账号注销流程

### 5.3 验收要求

- 首次登录能够创建用户。
- 再次登录能够识别同一用户。
- 登录后返回原业务入口。
- 401 后不会丢失未提交选择或考试草稿。
- 不同用户无法读取彼此的进度、错题、收藏和试卷。
- AppSecret 和数据库密码不出现在客户端包中。

微信登录参考：[微信小程序登录官方文档](https://developers.weixin.qq.com/miniprogram/dev/framework/open-ability/login.html)。

## 7. 500 道题库正式复核

自动校验只能证明数据结构合法，不能替代人工内容审核。正式发布前必须由非出题人员完成逐学科交叉复核。

### 7.1 复核内容

- 正确答案是否准确、完整。
- 题干是否存在歧义或缺少适用前提。
- 多选题是否遗漏正确选项。
- 干扰项是否合理。
- 解析是否说明原因而不是简单复述答案。
- 难度 1/2/3 是否真实。
- 15% 难题是否真正达到难题标准。
- C/C++ 和 STL 是否符合声明的语言标准。
- 是否依赖未定义行为或实现相关行为。
- 操作系统、网络和组成原理术语是否符合团队课程口径。
- 408 题目是否适合模拟和复盘。

### 7.2 复核记录

必须在 `content/REVIEW_STATUS.md` 中填写：

- 交叉复核人。
- 交叉复核日期。
- 交叉复核结论。
- 修改记录。

题库修改后必须执行：

```powershell
npm run build:data
npm run verify
```

## 8. 测试、安全与质量门禁

### 8.1 自动测试

- 普通答题和交卷幂等测试。
- 408 超时自动交卷测试。
- 题目版本快照测试。
- 数据备份和恢复测试。

### 8.2 故障场景

- 断网和弱网。
- 请求超时。
- 401、403、404 和 5xx。
- 重复点击和重复提交。
- 服务重启和数据库短暂不可用。
- 草稿保存失败和恢复。
- 到期交卷并发请求。
- 空题库、空错题和空收藏。

### 8.3 安全要求

- 全部 API 使用 HTTPS。
- 所有请求执行身份认证和用户级授权。
- 所有输入执行白名单和 Schema 校验。
- 登录和敏感接口执行限流。
- 防止通过会话 ID 读取其他用户数据。
- 防止待答题接口返回答案和解析。
- 日志不得记录 Token、AppSecret、数据库密码等敏感信息。
- 密钥统一使用服务器环境变量或密钥管理服务。
- 数据库定期备份并验证恢复可用性。
- 建立异常告警、事故处理和版本回滚方案。
